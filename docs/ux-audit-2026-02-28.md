## Findings by surface

### `/` (Home)
What’s strong:
- Strong visual identity, hierarchy, CTA clarity.
- Messaging is operator-focused and differentiated.

Gaps:
- No social proof (metrics are generic, not evidence-backed).
- CTA destination split (waitlist vs ops) could dilute first-time conversion.

### `/ops` (Operator queue)
What’s strong:
- Full end-to-end workflow present.
- Failure/retry/audit visibility exists.
- Workspace visibility fixed.

Gaps:
- Raw scaffolding UI (inline styles, dense forms, weak hierarchy).
- No loading/pending/success/error feedback per action.
- Too much on one page; cognitive load is high.
- No persistent “current step” or progress guidance.

### `/waitlist`
What’s strong:
- Good visual polish and concise pitch.
- Low-friction signup and clear CTA.

Gaps:
- No explicit privacy/reply expectations near form.
- Could use stronger trust elements (who this is for, expected timeline).

### `/waitlist/admin`
What’s strong:
- Practical filters + status management + export/copy.

Gaps:
- Table UI is utilitarian and heavy.
- No sticky controls / row feedback after status save.
- Large data handling could become painful without pagination/sorting.

---

## Priority punch-list

## P0 (do now)
1. Add action feedback states in `/ops` and `/waitlist/admin`:
   - pending buttons, inline success/error messages, recoverable errors.
2. Refactor `/ops` into step cards with stronger hierarchy:
   - Inputs grouped by stage, compact rows, visual step progress.
3. Add lightweight design tokens/components for shared controls:
   - Button/Input/Card/Badge, consistent radius/spacing/type scale.

## P1 (next)
4. Add pagination/sort/search to `/waitlist/admin` table.
5. Add confirmation/guardrails for high-impact actions (retry all, publish run).
6. Add trust copy near waitlist form (response SLA, privacy line).

## P2 (after)
7. Add small “activity timeline” component in `/ops` for recent actions/outcomes.
8. Add conversion proof section on `/` when real metrics are available.

Do **one focused UX sprint** before calling DO "premium":
